# FrameSmith Architecture

## What Is FrameSmith?

FrameSmith is a timeline engine.

It takes:

• A timeline made of tracks and clips
• Some clips backed by real media files
• Some clips generated by logic (like text overlays)

And it turns that into:

• Raw video frames
• Raw audio data
• Visual instructions for overlays

Later, those will be composed and exported.

---

# The Big Picture

FrameSmith is built as stages.

Each stage has one job.
Each stage hands its output to the next.
No stage cheats.

## Demux Baseline (Locked Decision)

Date locked: February 13, 2026

Current operating decision for timeline export debugging:

- Video demux in timeline defaults to `mp4box` (`script.js` `videoDemuxer` default).
- Native demux remains available via query override for comparison (`?videoDemuxer=native`).

Why this is locked:

- Isolated compare (`scripts/demux-isolation-node.mjs`) shows native and mp4box return the same video payload bytes/count, but different video timing/key semantics.
- Native demux currently reports all video samples as key and flattens timing to CFR-like deltas.
- mp4box reports mixed key/delta and decode-order-aware timing (with reorder behavior visible in PTS deltas).

Implication:

- During timeline compiler stabilization, treat native demux video semantics as known-unreliable until fixed.
- Use mp4box video demux as the baseline to reduce problem surface to timeline decode/compose/encode/mux behavior.

## Native Demux Remediation (Required)

Status: confirmed issue, fix pending

Scope:

- This is a native demux semantic issue (video timing/key metadata), not a timeline composition issue.
- Timeline export currently defaults to mp4box video demux as a stabilization baseline.

Required fixes in native demux:

1. Video key semantics
- Stop reporting all video access units as key.
- Preserve true sync/key semantics from container-derived sample tables.

2. Video timing semantics
- Preserve decode/presentation timing semantics from source sample tables.
- Do not flatten reordered presentation into synthetic CFR-like timing semantics.

3. Semantic parity tests
- Add demux comparison regression tests (native vs mp4box on same fixture).
- Assert parity boundaries:
  - sample count
  - key/delta distribution
  - DTS monotonic properties
  - PTS reorder characteristics

References (comparison tools and evidence):

- `scripts/demux-isolation-node.mjs` (node-side native vs mp4box compare)
- `scripts/mp4box-isolation-demux.js` (browser-side isolated compare)
- `scripts/mp4box-isolation.html` (browser harness UI)

Acceptance criteria:

- Timeline export using native demux produces no visual stutter on the current reference source.
- Isolated demux comparison shows native semantic output behavior aligned with mp4box for required invariants.
- New regression tests fail on old behavior and pass on corrected behavior.

## What Is a PreRenderPlan?

It is a flat list of fragments.

Example:

```
{
  fragments: [
    { ACCESS_UNITS fragment },
    { PROCEDURAL fragment }
  ]
}
```

Planning:

• Reads timeline
• Splits clips into fragments
• Groups by contributor type

Planning does NOT:

• Decode
• Render
• Evaluate time

It just describes what exists.

## Execution Strategies (Current Direction)

This seam exists so multiple execution strategies can share one plan contract.

Planned strategy types:

• ExportExecutionStrategy
• PreviewExecutionStrategy

ExportExecutionStrategy will target deterministic offline output.

PreviewExecutionStrategy will target interactive playback behavior.

Both consume the same `PreRenderPlan` boundary and differ by execution policy.

# Stage 5 — Composition (Minimal Production Seam)

File:

```
src/composition/composeAtTime.js
```

Current contract:

```
composeAtTime({
  timeSeconds,
  decodedContainerBackedFragmentBatch,
  activeLayers,
  renderIntents,
  options
})
```

Current behavior:

• Composes one timeline time slice per call
• Orders layers by `zIndex`
• Returns deterministic composed output shape

Current output shape:

```
{
  composedVideoFrame: { timestamp, layerOrder, videoFrame },
  composedAudioData: { timestamp, audioStrategy, audioData },
  diagnostics: { issues: [] }
}
```

This seam is intentionally simple today.
Real pixel/audio composition can replace internals later without changing the contract.

# Memory Ownership and Future Optimization Seams

FrameSmith memory pressure is shared across three execution boundaries:

1. Composition (`composeAtTime`)
   • Allocates/attaches `VideoFrame` and `AudioData`
   • Must support artifact lifetime policy changes later

2. Encoding (`encodeAtTime`)
   • Controls per-slice encode handoff
   • Is the seam for backpressure and queue policy changes

3. Export orchestration (`ExportExecutionStrategy.execute`)
   • Owns loop retention policy (what is kept vs dropped each step)
   • Is the seam for bounded pipelines later

NativeMuxer has its own memory characteristics (payload storage and `mdat` assembly),
but FrameSmith must remain structured so composition/encode/orchestration can evolve
independently before bytes cross the mux boundary.

Current policy:

• Keep architecture seams explicit
• Avoid premature optimization in stage internals
• Preserve contracts so memory policies can be swapped in without boundary churn

# MVP Export Constraint (WebCodecs Happy Path)

Current FrameSmith export intentionally constrains video encoding to a simple
WebCodecs shape aligned with NativeMuxer semantic end-to-end coverage:

• Realtime video encoder configuration
• Forced all-intra output (`keyFrame: true` for each encoded frame)
• Stable CFR timestamp grid from export cadence

Why this constraint exists now:

• Timeline MVP must stay on a known-good mux input contract
• NativeMuxer policy work for richer sync/composition signaling is still evolving
• We avoid relying on reorder-sensitive metadata paths during MVP (for example CTTS)

What this means:

• Larger files than long-GOP output
• Lower compression efficiency
• Better determinism and lower timing ambiguity in current browser playback path

Planned evolution after MVP:

• Allow long-GOP / reordered encode output
• Emit and validate sync/composition timing representation consistently (`stss`/`ctts`)
• Keep this behind the existing adapter seam (`adaptEncodedOutputsToMp4BuildInput`)
  so timeline/composition architecture does not churn

Because:

• Each stage is small
• Each stage has one job
• Each stage can be tested independently
• Names encode scope

You can change the decode strategy
without touching the planner.

You can change composition
without touching procedural resolution.

That is real separation.
